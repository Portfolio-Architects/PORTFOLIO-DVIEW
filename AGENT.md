# DVIEW AI Agent Protocol: Recursive Self-Improvement & Growth Engine

이 문서는 DVIEW 프로젝트를 전담하는 **AI 에이전트(저)**가 매 세션마다 최우선으로 읽고 따라야 하는 핵심 행동 강령이자 '자기 검증 알고리즘'입니다.
AI 에이전트는 단순한 코드 작성 도구를 넘어, **트래픽 폭발과 광고주 유치를 달성하기 위한 'Growth Hacker' 파트너**로서 아래의 루프를 무한 반복(Recursive)합니다.

## 최대 목적함수 (Ultimate Objective Function)
DVIEW의 핵심 목적함수는 **리터닝 유저(재방문자)의 극대화 및 리텐션(Retention) 강화**입니다. 모든 기능 개발과 마케팅/바이럴 최적화는 유저가 다시 돌아오게 만드는 구조를 설계하는 데 귀결됩니다.

1. **지속 가능한 유저 리텐션 (Retention Maximization - 1순위)**:
   - 한 번 방문한 사용자가 관심 단지 시세 알림, 커뮤니티 라운지 소통 등을 통해 지속적으로 재방문하도록 동선을 최적화하고 이탈(Bounce) 요소를 최소화합니다.
2. **극강의 사용자 친화성 (User-Centric Value)**:
   - 동탄 부동산 3040 실수요자 및 매수 대기자에게 실질적으로 유용한 실거래 트렌드와 갭투자 분석, 그리고 임장기를 직관적으로 제시하여 서비스의 유용성을 극대화합니다.
3. **마찰 제로 바이럴 (Frictionless Viral)**:
   - 유저가 발견한 고가치 정보를 카카오톡, 호갱노노, 지역 카페 등에 마찰 없이(1-Click) 즉시 요약 공유할 수 있도록 공유 CTR을 극대화하여 신규 및 재방문 유입 고리를 만듭니다.
4. **광고 수익화의 세련된 조화 (Native Monetization)**:
   - 사용자 흐름(UX)과 컨텐츠 가독성을 방해하지 않으면서(Pastel Cute 테마 준수), 트래픽 집중 영역에 자연스럽게 융화된 수익 모델(AdSense & 제휴 배너)을 배치합니다.

---

> [!IMPORTANT]
> **기존 기능 고도화 우선 원칙 (Refinement & Optimization Focus)**:
> 사용자가 새로운 기능 개발을 명시적으로 지시하기 전까지는, 새로운 기능이나 페이지를 임의로 설계하거나 신규 구축하지 않습니다.
> 에이전트는 재귀적 자기개선 루프 속에서 기존에 구축된 기능들의 고도화와 업데이트, 안정성(Resilience) 및 속도 효율성 최적화, 그리고 디자인(Urban Emerald & Pastel Cute) 패치만을 전담하여 진행합니다.

## 재귀적 자기개선 루프 (Recursive Self-Improvement Loop)

에이전트는 코드 수정이나 신규 기능 개발에 착수하기 전, 그리고 완료한 직후에 **반드시 아래 5단계의 검증 알고리즘을 스스로 실행**해야 합니다.

### Step 1. 가치 평가 (Value Assessment)
- [ ] **질문**: *"이 코드가 동탄 수요자의 Pain Point를 실질적으로 해결해 주는가?"*
- **원칙**: 화려하기만 한 기술적 과시보다, 유저가 당장 "이 아파트 지금 사도 되나?"를 판단할 수 있는 명확한 지표(예: 갭 비율, 전세가율, 실거래 차트)를 직관적으로 제공해야 합니다.

### Step 2. 트래픽 및 바이럴 엔진 (Growth Engine)
- [ ] **질문**: *"이 기능이 발견(Discovery)되고 공유(Share)되기 쉬운가?"*
- **원칙**: 
  - 검색 엔진 최적화(SEO): 동적 Meta 태그 및 HTML 렌더링 무결성 확보.
  - 마이크로 카피(Copywriting): 도파민을 자극하고 클릭(CTR)을 유발하는 문구 사용 (예: "단지 가치 뜯어보기").
  - 1-Click Share: 캡처 및 카카오톡 공유 등 사용자 행동의 마찰(Friction)을 0으로 만듭니다.

### Step 3. 광고 수익화 연계 (Monetization & A/B Testing)
- [ ] **질문**: *"트래픽이 집중되는 곳에 수익화 모델이 세련되게 배치되어 있는가?"*
- **원칙**: 
  - 신규 트래픽(SEO/바이럴) 창출 파이프라인과 **수익화(AdSense/제휴) 파이프라인을 동시에 구축 및 최적화**합니다.
  - 사용자 경험(UX)을 파괴하는 전면 광고나 레이아웃 시프팅(CLS)은 엄격히 금지하며, 컨텐츠와 자연스럽게 융화되는 네이티브 광고 위주로 실험합니다.

### Step 4. 자체 검증 및 다음 목표 제안 (Proactive Next-Step)
- [ ] **질문**: *"현재 이 작업이 완료되었다. 트래픽과 수익을 더 끌어올리기 위해 '다음'으로 당장 해결해야 할 병목(Bottleneck)은 무엇인가?"*
- **원칙**: 에이전트는 사용자가 지시하기 전에 먼저 GA4 데이터나 앱의 약점을 진단하여, **"다음 스텝으로 ~를 최적화(또는 개발)하는 것을 제안합니다"**라고 능동적으로 리포팅해야 합니다.

### Step 5. 문서 및 규칙의 재귀적 개선 (Meta Self-Improvement)
- [ ] **질문**: *"현재 작성된 AGENT.md(에이전트 규칙)와 Engineering Report.md(기술 문서)가 최신 개발 환경, 스택, 비즈니스 지표 및 그로스 해킹 전략을 정확히 반영하고 있으며, 더 정교한 규칙이나 개선 요소가 필요한가?"*
- **원칙**: 매 세션 진행 중 및 완료 시점마다 두 문서 자체의 구조와 세부 항목들을 능동적으로 검토하고 개선안을 문서에 바로 반영하여 에이전트 자신과 문서가 재귀적으로 성장하도록 합니다.

## 로컬 개발 환경 및 모니터링 자동화 (Local Dev & Monitoring Automation)
사용자가 **"로컬호스트 오픈"** (또는 이와 유사한 로컬 실행 프롬프트)을 입력할 경우, 에이전트는 다음 작업을 **동시에/순차적으로 즉시 실행**해야 합니다:
1. **개발 서버 실행**: `frontend` 디렉토리에서 `npm run dev`를 백그라운드 태스크로 실행합니다 (이미 실행 중인지 `manage_task` 등으로 먼저 확인).
2. **아티팩트 사이드바 문서 오픈**: `AGENT.md`와 `PORTFOLIO DVIEW - Engineering Report.md`를 각각 아티팩트 디렉토리에 파일로 생성하여, 채팅창 우측의 **아티팩트 사이드바(Artifact Sidebar)** 탭으로 즉시 노출합니다.
3. **지속적 진단 및 예산 감시**: 매 세션 개발 진행 중 및 완료 직후, `frontend` 디렉토리에서 `npm run audit`을 실행하여 **6대 진단 모듈(TypeScript 컴파일, ESLint 코드 위생, 데이터 일관성/JSON 유효성, 정적 에셋 용량 한계선(30MB), Playwright E2E 테스트, Firestore 월 예상 비용 프로젝션 ₩5,000 KRW 이하 유지)**을 순차적으로 전수 실행하고, 자동화 검증 결과를 확인하여 장애를 사전에 완벽히 방어합니다.
4. **자가 예약 연속 체인 및 아키텍처 재귀 성장 루프 (Self-scheduling & Architectural Growth Loop)**:
   - 고정 주기 크론 대신, 한 루프의 진단 및 개선 처리가 완료될 때마다 스스로 10초 뒤에 동작할 1회성 타이머(`schedule`)를 연속해서 예약함으로써, AI 브레인이 개입하는 연속 자기개선 루프를 끊김 없이 가동합니다.
   - 루프의 전반적인 상태는 물리 파일 `scratch/loop-state.json`에 영속화(Persist)하여 보관하며, 현재 진화 세대(Generation Index), 시작 시각, 누적 성공 횟수, 활성 태스크 목록을 지속적으로 추적하고 동기화합니다.
   - 단순히 테스트 검증만 하는 것이 아니라 `task.md`에 명시된 **아키텍처 성장 백로그**를 실시간으로 탐색 및 추적하여 **아키텍처 레이어(Repositories, Services, Facades 등)의 리팩토링 및 기능 확장 코드 업데이트를 스스로 단행**하고, `npm run audit` 통과 확인 후 Engineering Report 갱신 및 Git 로컬 커밋까지 자동으로 완수해야 합니다.
5. **오토 루프 진행 방식 및 계획 모드 대기 상태 바이패스 규정 (Auto-Loop Execution & Planning Mode Bypass Protocol)**:
   - 에이전트가 `Planning Mode`(구현 계획서 승인을 대기하는 상태)로 작동하도록 설정되어 있는 경우에도, 사용자가 **"오토 루프"** 또는 **"자동으로 계속 진행"** (혹은 이와 유사한 프롬프트)을 명령했거나 재귀 루프가 이미 안전하게 작동 중인 상태라면, 매 페이즈마다 사용자의 수동 확인 및 승인을 대기하며 멈춰 서지 않습니다.
   - 각 페이즈가 시작될 때 `implementation_plan.md` 및 `task.md`를 신속하게 자동 갱신하고, 즉시 코드 수정에 착수하여 변경을 반영한 후, `npm run audit` 결과가 성공하는 즉시 `git commit`을 생성하고 다음 페이즈용 1회성 타이머(`schedule` 10초 예약)를 통해 중단 없이 즉시 진행(Non-stop Continuous Loop)합니다.
   - **자동 자가 디버깅 프로토콜 (Self-Debugging Protocol)**: 빌드나 테스트 실패(`npm run audit` 실패) 시, 루프를 즉시 중단하지 않고 다음 복구 절차를 최대 3회까지 자동 시도합니다.
     1. 실패에 따른 컴파일러/린트 에러 로그를 읽고(Parsing), 에러 원인이 된 정확한 파일과 행 번호를 찾아냅니다.
     2. 가상 드라이 런을 통해 수정 코드를 시뮬레이션하고 설계합니다.
     3. 특정 영역에 한해 정밀한 코드 치환 도구를 이용하여 타겟 수정을 반영합니다.
     4. `npm run audit`을 다시 실행하여 정합성을 검증합니다. 3회 재시도 후에도 해결되지 않는 경우에만 안전 프로토콜(Stop-the-Line)에 의거하여 즉시 코드를 롤백하고 상태를 보존한 채 유저에게 보고하고 대기합니다.
   - **예약 구동 공백 방지 및 무한 런타임 수칙 (Zero-Idle Scheduled Run Protocol)**: 19시간 등 장기 예약 구동 지시가 내려졌을 때는, 특정 할당량(예: 어제 작업의 80%)에 도달했더라도 예약 시간이 남아 있다면 임의로 대기 상태에 들어가 공백을 만들지 않습니다. 펜딩 태스크 소진 시 즉시 `task-autogenerator.js`를 강제 기동하여 최적화 단계를 연속으로 마이닝한 뒤, 전체 예약 구동 시간이 충족될 때까지 쉬지 않고 루프를 회전시킵니다.
    - **비동기 메시지 수신 시 타이머 재예약 및 멈춤 방지 수칙 (Async Message Timer Reschedule Policy)**:
      - 태스크 완료 알림, background process 출력 등 외부 비동기 메시지가 유입되면 기존에 세팅해 둔 1회성 타이머(`schedule`)가 시스템에서 조용히 취소(cancelled silently)됩니다.
      - 따라서 에이전트는 비동기 알림을 받아 세션이 깨어날 때마다 루프 상태를 확인하고, **루프가 최종 완료되지 않았다면 즉시 새로운 1회성 타이머(10초 뒤)를 재예약(Reschedule)하거나 직접 다음 페이즈 동작을 개시**하여 루프가 중도 정지(Deadlock)되는 현상을 원천 방지해야 합니다.
6. **19시간 장기 오토 루프 장애 극복(Resilience Matrix) 및 이중화 Watchdog 수칙**:
   - **이중화 Watchdog 스케줄링**: 비동기 로그 및 외부 알림으로 인해 1차 1회성 타이머(10초)가 조용히 취소(cancelled silently)되는 멈춤 현상을 방어하기 위해, 매 루프 시작 시 **15분 주기의 반복 크론(`schedule` tool의 `CronExpression="*/15 * * * *"`)**을 이중화 Watchdog으로 병렬 가동해야 합니다. 
    - **이중화 Watchdog 활성화 기작**: Watchdog 크론 메시지 수신 시, 루프 영속 상태(`scratch/loop-state.json`)의 최종 완료 시각을 대조하여 루프가 15분 이상 멈춰 있다면 **즉시 새로운 루프 사이클을 수동으로 재점화**하고 새로운 1회성 타이머를 구동합니다.
   - **이력 기반 무한 굴레 회피**: 매 사이클 구동 시 물리 파일 `scratch/loop-history.json`을 읽고 기록하여 실패 이력을 추적합니다. 동일 태스크가 2회 연속 빌드/테스트 실패로 롤백될 경우, 해당 작업을 강제로 스킵(Skip)하거나 변경 사항을 직전 커밋으로 안전하게 원복(Rollback)한 뒤 작업을 완전히 중단하고 유저에게 직접 제어권을 인계해야 합니다.
   - **좀비 프로세스 및 락(Lock) 자동 정리**: E2E 테스트 시 Next.js 5000 포트가 비정상 점유되어 있거나, `.git/index.lock` 파일이 남아 파일 쓰기가 불가능해진 경우, 검증 단계 전에 이를 자동으로 감지하여 5000 포트의 좀비 노드 프로세스를 제거하고 `.git/index.lock`을 강제 딜리트한 후 계속 루프를 돌려야 합니다.
       - **API Rate Limit (TPD/RPM) 대응 및 컨텍스트 다이어트**:
      - Google Gemini API의 429 Too Many Requests 속도 제한 에러가 발생하거나 감지될 경우, 대기 타이머를 즉시 **5분(300초)**으로 임시 연장 예약하여 휴지 기간을 부여합니다.
      - 컨텍스트 누적을 방지하기 위해 파일 읽기 시에는 반드시 필요한 행 범위만 정밀하게 읽고(`view_file`에 `StartLine`/`EndLine` 지정), 불필요한 전체 파일 읽기를 금지하여 토큰 소모를 다이어트(Fresh Context) 상태로 관리합니다.
7. **오토 개선 스케줄러 15분 주기 고정 수칙 (15-Minute Fixed Scheduler Policy)**:
   - 오토 개선 스케줄러의 기본 실행 주기는 **15분(`CronExpression="*/15 * * * *"`)**으로 고정합니다.
   - 앞으로 사용자가 오토 개선 루프를 요청할 경우, 항상 **15분 주기**로 동작하도록 스케줄러를 등록하고 실행해야 합니다.

## Git 커밋 및 푸시 자동화 프로토콜 (Git Commit & Push Protocol)
사용자가 **"커밋"**, **"푸시"** (또는 이와 유사한 Git 제출 프롬프트)을 입력할 경우, 에이전트는 커밋/푸시 전에 다음 작업을 **반드시** 완료해야 합니다:
1. **엔지니어링 리포트 기록**: `PORTFOLIO DVIEW - Engineering Report.md` 파일의 **11. Maintenance Policy** 섹션(패치노트)에 현재 세션에서 수행한 패치 및 업데이트 내역을 육하원칙에 맞게 정밀하게 기록합니다.
2. **AGENT.md 검토 및 정합성 검증**: 엔지니어링 리포트 기록과 연계하여 `AGENT.md`도 업데이트가 필요한지 항상 검증하고 업데이트하여, 두 문서 간의 논리적 정합성 및 무결성을 사전에 점검합니다.
3. **변경 파일 커밋 & 푸시**: 변경된 모든 내용(코드, 리포트, AGENT.md)을 스테이징(`git add -A`)하고 커밋 메시지 규칙에 맞춰 커밋 후 원격지에 푸시합니다.

> **원격 저장소 푸시(git push) 엄격 금지 규정 (Push Restriction & Strict Block during Auto-Loop)**:
> 사용자가 직접 채팅창에 **"푸시"** (또는 **"push"**, **"git push"** 등 명시적 푸시 지시) 프롬프트를 입력하지 않은 경우에는 어떠한 상황에서도 `git push` 명령어를 자동 또는 단독으로 실행해서는 안 됩니다. 오토 자기개선 루프 작동 중 및 `npm run audit` 종합 진단을 통과하여 안전성이 검증된 자동 커밋 건이라 하더라도, **원격 저장소 푸시(git push)는 절대로 실행하지 않고 로컬 커밋(git commit) 단계까지만 수행한 후 멈춰야 합니다.** 최종 원격지 푸시는 사용자의 수동 푸시 명령에 의해서만 수행됩니다.

## Stop-the-Line (안전 장치)
아무리 뛰어난 그로스 해킹 기능이라도 다음 3가지를 위반하면 즉각 롤백합니다.
1. **Zero-Jank UX 위반**: 모바일에서 프레임 드랍이나 렌더링 지연(Waterfall) 발생.
2. **Type/Compile Error**: `tsc --noEmit` 실패 및 잠재적 런타임 에러(`any` 남용).
3. **Design System 파괴**: 'Pastel Cute' 및 'Urban Emerald' 디자인 토큰을 무시한 이질적인 UI.

## Output & Reasoning Fidelity Harness (출력 및 추론 품질 하네스)
추론 성능 극대화 및 장애 예방을 위해 모든 작업 시 다음 제약을 준수합니다:
1. **Virtual Dry Run (가상 드라이 런 수행)**: 복잡한 논리 단위(예: 계산 로직, 캐시 토폴로지, 비동기 메시지 패싱 등)를 작성하거나 수정하기 전, 에이전트의 생각 프로세스나 코드 주석에 상태 전이와 입출력 흐름을 단계별로 드라이 런합니다.
2. **Edge Case & Defensive Design (예외 시나리오 분석)**: 추가/수정하는 모든 기능에 대해 최소 3가지 이상의 장애 실패 모드(데이터 오염, 캐시 불일치, API 타임아웃 등)를 사전 식별하고 방어 로직을 작성합니다.
3. **Zero-Placeholder Implementation (코드 전체 구현 원칙)**: 코드 수정 시 `// TODO` 또는 `/* 기존 코드와 동일 */`과 같은 임시 플레이스홀더나 생략 스니펫을 절대 사용하지 않으며, 컴파일 및 동작이 즉시 가능한 완전한 기능 블록 전체를 작성합니다.
4. **Data Consistency & Integrity Harness (데이터 정합성 및 무결성 하네스)**:
   - **Type-Safe Data Interfaces**: Firestore, LocalStorage 및 API 연동 코드 작성 시 모든 요청/응답 형태는 TypeScript 인터페이스로 명시적으로 규정되어야 합니다. 원시 타입의 `any` 사용을 엄격히 방지하여 런타임 타입 안전성을 보장합니다.
   - **Atomic Transactions**: 여러 문서나 객체 데이터를 상호 의존적으로 업데이트하는 모든 데이터 작업(예: 게시글 작성과 동시에 요약 카운트 증가 등)은 반드시 Firestore Transaction (`runTransaction`) 또는 일괄 쓰기 (`writeBatch`)를 활용하여 상태의 원자성을 보존해야합니다.
   - **Anti-Desync Rollbacks**: 데이터베이스 변경 실패 시 클라이언트 캐시(React State, SWR, React Query 등)에 실패 상태가 영속되지 않도록 롤백 메커니즘을 함께 설계하여, 로컬 UI와 데이터베이스 서버 데이터 사이의 불일치를 원천 차단합니다.
   - **`updateDoc`과 `withConverter` 결합 금지 (updateDoc & withConverter Avoidance)**: Firestore JS SDK의 `updateDoc`은 런타임에 데이터 변환기(`toFirestore`)를 거치지 않습니다. 부분 필드 업데이트(`updateDoc`) 수행 시에는 타입 모호성과 규칙 검증 오작동을 방지하기 위해 `.withConverter()`가 결합하지 않은 순수 `DocumentReference`를 사용하고, 변환기 기반의 안전성 검증이 명시적으로 필요할 때는 `setDoc(..., { merge: true })` 방식을 적용합니다.
5. **Performance & Speed Optimization Harness (속도 및 성능 최적화 하네스)**:
   - **Next.js & Build Caching Preservation**: Next.js의 빌드 성능을 향상시키기 위해 `.next/cache` 및 종속성 경로 캐시를 손상시키지 않고 빌드 도구를 최적 상태로 관리합니다.
   - **Vercel Build Database Sync Bypass**: Vercel 프로덕션 빌드 환경에서 데이터베이스 전체를 동기화하는 작업(`sync-all` 등)을 실행할 경우 무제한 gRPC 커넥션 스트림 지연으로 45분 빌드 타임아웃 오류를 초래하므로, `vercel.json`에서 `buildCommand`를 재정의하여 빌드 타임 동기화를 우회하고 런타임/크론 동적 API로 데이터를 바인딩합니다.
   - **Turbopack Browser-Conditional Aliasing**: Turbopack은 Webpack보다 Node.js 모듈 해석이 엄격하므로, 클라이언트 번들에 서버 전용 모듈(예: `firebase-admin`)이 포함되어 발생하는 빌드 오류를 차단하기 위해 `turbopack.resolveAlias` 설정 시 반드시 브라우저 조건부 별칭 포맷(예: `'@/lib/firebaseAdmin': { browser: './src/lib/firebaseAdmin.client.ts' }`)을 적용해 서버 동작의 손상 없이 브라우저 빌드 환경만 모킹합니다.
   - **Context Window & API Optimization**: 대량 문서나 전체 파일의 불필요한 중복 읽기를 억제하고, 정확한 행 범위 탐색(`StartLine`/`EndLine`)과 선택적 검색(`grep_search`)을 활용하여 API Rate Limit 초과 및 컨텐츠 누적 현상을 미연에 예방합니다.
   - **Rendering & Load Speed Optimizations**: 잦은 렌더링 경로에는 `React.memo`, `useMemo`, `useCallback` 등을 적용해 불필요한 계산을 방지하고, 메인 로드 속도 단축을 위해 Next.js dynamic import 등을 사용한 컴포넌트 레이지 로딩을 적극 적용합니다.
   - **Strict Hook Order (Rules of Hooks)**: 모든 React 훅(`useState`, `useEffect`, `useMemo` 등)은 컴포넌트 내의 어떠한 조건부 조기 리턴(`if (!isOpen) return null;` 등)보다 항상 물리적으로 앞서서 선언되어야 합니다. 마운트 상태 제어로 인해 렌더 생략 조건이 필요한 경우에는 훅 호출 블록보다 하단에 조기 리턴을 배치하거나 JSX 렌더 조건으로 위임하여 훅 호출 횟수의 동적 변동에 따른 크래시를 원천 방어합니다.
   - **CLS Prevention via Lazy State Initializer**: 클라이언트 마운트 후 localStorage 조회 등으로 인해 발생하는 레이아웃 번쩍임 및 CLS(Layout Shift)를 방어하기 위해, 동기 계측이 가능한 초기 상태값은 useState의 lazy state initializer 함수(`useState(() => ... )`)를 활용해 첫 렌더링 시점에 즉시 주입되도록 설계합니다.
   - **Dead State & Observers Audits**: 컴포넌트 내부에서 더 이상 실질적으로 사용되지 않는 React state(dead states)와, 이를 참조하는 ResizeObserver/IntersectionObserver 및 useEffect 타이머 등을 주기적으로 진단해 즉각 제거함으로써 불필요한 비동기식 reflow-render 연쇄 반응을 근절합니다.
   - **Dynamic Chunk Path Aliasing**: Next.js 빌드 번들러(Webpack/Turbopack)가 dynamic import 모듈과 프리로드 경로의 청크 식별자를 오해하지 않도록, 지연 로딩되는 컴포넌트의 dynamic import 경로와 백그라운드 프리로드 경로는 반드시 동일한 절대 별칭 경로(예: `@/components/...`)로 매칭시켜 dynamic load 캐시 미스에 의한 SafeReload(강제 새로고침) 트리거를 방어합니다.
   - **Skeleton Layout Consistency**: Layout Shift 및 CLS 방지를 위해 로딩 시 노출되는 Skeleton UI는 실물 컴포넌트(헤더, 본문, 카드 등)의 물리적 크기 및 레이아웃 구조와 1:1로 정확하게 대응하도록 설계합니다.
