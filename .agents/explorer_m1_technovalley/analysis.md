```typescript
  { name: '바이오·헬스케어', value: 1.8, color: '#f59e0b', ... }, // Amber
  { name: '지식기반 서비스', value: 21.7, color: '#fdba74', ... }, // Light Peach
  { name: '정밀기기 및 기타', value: 33.7, color: '#e7e5e4', ... }  // Warm Gray

```

### Current Hover/Scale Handling
Currently, the chart relies on **click states** (`activeCategory`) rather than hover states to emphasize slices. When a slice is clicked, `activeCategory` is updated, causing:
1. `stroke` to draw a prominent border around the active slice: `stroke={isSelected ? 'var(--text-primary)' : 'var(--bg-surface)'}`.
2. `strokeWidth` to increase: `strokeWidth={isSelected ? 4 : 2.5}`.
3. `opacity` to dim the non-selected slices: `opacity={activeCategory === null || isSelected ? 0.99 : 0.6}`.

There is **no hover transition or scaling** applied to the slices, which can make the chart feel static.

### CSS/SVG Transition Optimization Proposal
To add a scale-out highlighting effect on hover *without* triggering JavaScript-based state updates (which cause heavy CPU calculation, React virtual DOM re-renders, and browser reflow rendering lag), we should apply **pure CSS animations** to the SVG paths generated by Recharts `<Cell>` components.

#### Proposed Code Modification
Under the `donutData.map` loop (lines 929–942):

**Before:**
```typescript
{donutData.map((entry: any, index: number) => {
  const isSelected = activeCategory === entry.name;
  return (
    <Cell 
       key={`cell-${index}`} 
       fill={entry.color} 
       stroke={isSelected ? 'var(--text-primary)' : 'var(--bg-surface)'}
       strokeWidth={isSelected ? 4 : 2.5}
       opacity={activeCategory === null || isSelected ? 0.99 : 0.6}
       style={{ outline: 'none', cursor: 'pointer' }}
       onClick={() => setActiveCategory(isSelected ? null : entry.name)}
    />
  );
})}
```

**After (Optimized with CSS Transitions):**
```typescript
{donutData.map((entry: any, index: number) => {
  const isSelected = activeCategory === entry.name;
  return (
    <Cell 
       key={`cell-${index}`} 
       fill={entry.color} 
       stroke={isSelected ? 'var(--text-primary)' : 'var(--bg-surface)'}
       strokeWidth={isSelected ? 4 : 2.5}
       opacity={activeCategory === null || isSelected ? 0.99 : 0.6}
       className="transition-all duration-300 transform hover:scale-[1.03] origin-center focus:outline-none cursor-pointer"
       style={{ 
         outline: 'none',
         transformOrigin: '50% 50%',
         willChange: 'transform'
       }}
       onClick={() => setActiveCategory(isSelected ? null : entry.name)}
    />
  );
})}
```
*Note: Using `scale-[1.03]` instead of `scale-105` prevents the scaled slice from clipping against the edges of the parent SVG element, and `transformOrigin: '50% 50%'` ensures that each SVG path scales correctly outward relative to the center of the Pie layout.*

## 4. Trend Graph (LineChart) Curve & Responsiveness

* **File Path**: `frontend/src/components/macro/TechnoValleyDashboard.tsx`
* **Main Dashboard LineChart Range**: Lines 1260–1353
* **Modal LineChart Range**: Lines 1647–1720

### Curve Interpolation Adjustment
The lines are configured with `type="monotone"`, which creates a smooth curve but can sometimes cause overshoot on steep changes. Changing the curve type to `natural` generates a natural cubic spline interpolation, which is visually pleasing and fits continuous trend representations.

#### Line Changes (Monotone to Natural)
We should modify the `Line` elements in both the main graph and the detail modal:

* **In the Main Dashboard (lines 1290-1336)**:
  ```typescript
  // For representative buildings:
  <Line type="natural" dataKey={b.id} ... />
  // For average line:
  <Line type="natural" dataKey="평균공실률" ... />
  // For rent comparison:
  <Line type="natural" dataKey={b.rentKey} ... />
  <Line type="natural" dataKey="평균임대료" ... />
  ```

* **In the Comparison Modal (lines 1668-1715)**:
  ```typescript
  // For selected buildings:
  <Line type="natural" dataKey={b.id} ... />
  <Line type="natural" dataKey="평균공실률" ... />
  // For rent comparison:
  <Line type="natural" dataKey={b.rentKey} ... />
  <Line type="natural" dataKey="평균임대료" ... />
  ```

### ResponsiveContainer Size Verification
The Recharts `ResponsiveContainer` wrapping the LineCharts has the following configurations:
- **Main Graph** (Line 1263): `<ResponsiveContainer width="100%" height="100%" minWidth={0} minHeight={0}>`
- **Modal Graph** (Line 1648): `<ResponsiveContainer width="100%" height="100%" minWidth={0} minHeight={0}>`

* **Why this is correct**:
  By default, `ResponsiveContainer` tries to calculate its size based on its parent container. However, if the parent has flex styles or is in a CSS Grid (such as the `lg:col-span-6` grid wrapper here), the container can fail to shrink because it reports its inner SVG width as its minimum width. Adding `minWidth={0}` and `minHeight={0}` forces the container to allow shrinking to zero, preventing container layout overflows on window resize. No action is required as this fix is already in place.

## 5. Mobile Touch Sensitivity, Momentum Scrolling, & Paddings

### Mobile Touch Sensitivity
- Checked `frontend/src/app/globals.css` (line 128):
  ```css
  button, a, [role="button"], summary, select {
    cursor: pointer;
    touch-action: manipulation;
  }
  ```
  - **Effect**: Explicitly sets `touch-action: manipulation`, which disables double-tap zoom gestures on modern mobile browsers. This completely removes the 300ms click delay, making tap actions instantly responsive.

### Scrolling Momentum
- iOS devices require `-webkit-overflow-scrolling: touch` for physics-based inertial scrolling.
- Checked `globals.css` (line 228):
  ```css
  .custom-scrollbar {
    scrollbar-width: thin;
    scrollbar-color: #d1d6db transparent;
    overflow-x: hidden !important;
    -webkit-overflow-scrolling: touch;
  }
  ```
  - Since the company card grid container uses `.custom-scrollbar` (line 1474), it automatically inherits scrolling momentum on iOS!
  - **Critique & Optimization**: The modal body container (line 1600) uses `scrollbar-thin` but does not include `.custom-scrollbar`. It should be updated to use `.custom-scrollbar` to ensure momentum scrolling is also active inside the comparison modal.

### Card Margins and Paddings on Mobile
- The outer panel wraps use `p-6 rounded-[24px]` on mobile (e.g. lines 900, 1162, 1360).
- **Optimization suggestion**: A padding of `p-6` (24px) takes up significant space on small mobile viewports (such as 360px wide devices). We should optimize these wrappers to use responsive paddings:
  ```className
  className="... p-4 sm:p-6 rounded-[20px] sm:rounded-[24px]"
  ```
  This increases available workspace inside cards by 16px horizontally on mobile, ensuring charts and tables do not feel cramped.
