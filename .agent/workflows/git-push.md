# DTDLS Git Push 워크플로우

코드 변경이 완료된 후 사용자가 커밋 또는 푸시를 요청하면 다음 순서로 작업을 수행합니다.

## Step 1. 문서화 및 정합성 검토 (필수)

1. **엔지니어링 리포트 기록**:
   - `PORTFOLIO DRIVE - Engineering Report.md` 파일의 **11. Maintenance Policy** 섹션(패치노트) 맨 위에 현재 세션의 패치 및 업데이트 내역을 구체적으로 기록합니다.
   
2. **AGENT.md 검토 및 정합성 검증**:
   - 리포트 수정 내역과 연계하여 `AGENT.md` 파일도 업데이트가 필요한지 검증하고 필요 시 즉시 반영합니다.
   - 두 문서 간의 논리적 정합성 및 무결성이 유지되는지 최종 검토합니다.

## Step 2. Git 제출 진행

1. **변경 파일을 스테이징합니다**:
   ```bash
   git add -A
   ```
   작업 디렉토리: 프로젝트 루트 디렉토리

2. **변경 내역을 커밋합니다**:
   - 커밋 메시지는 변경 사항을 요약하여 다음 커밋 타입 가이드에 따라 작성합니다.
   ```bash
   git commit -m "<type>: <한글 설명>"
   ```
   작업 디렉토리: 프로젝트 루트 디렉토리

   커밋 타입 가이드:
   - `feat`: 새 기능 추가
   - `fix`: 버그 수정
   - `style`: 디자인/UI 변경
   - `refactor`: 코드 리팩토링
   - `docs`: 문서 변경

3. **원격 저장소에 푸시합니다 (명시적 푸시 프롬프트 필수)**:
   - **⚠️ 경고**: 사용자가 직접 채팅창에 **"푸시"** (또는 **"push"**, **"git push"**) 프롬프트를 입력하지 않은 경우에는 **절대로 `git push`를 실행해서는 안 됩니다.** 사용자가 "커밋"만 지시했거나 명시적으로 푸시를 요청하지 않은 경우에는 2단계(커밋)까지만 완료하고 대기해야 합니다.
   - 오직 사용자가 명시적으로 푸시를 입력한 경우에만 아래 명령어를 제안하고 실행합니다.
   ```bash
   git push
   ```
   작업 디렉토리: 프로젝트 루트 디렉토리
